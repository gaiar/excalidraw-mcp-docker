# Security

Security architecture for the self-hosted Excalidraw MCP server with Docker + Valkey.

## Threat Model

This deployment targets accidental exposure and opportunistic attacks, not sophisticated targeted threats. The primary threat surface is a misconfigured or over-exposed deployment on a home network or VPS.

| Threat                                                   | Mitigation                                                                     |
| -------------------------------------------------------- | ------------------------------------------------------------------------------ |
| Unauthorized access to checkpoint data (stored diagrams) | Valkey ACL user with key-pattern restrictions; no exposed port                 |
| Data exfiltration from Valkey                            | Internal Docker network; no host port binding; TLS in transit                  |
| MITM between app and Valkey containers                   | TLS encryption; optional client certificates; mandatory password auth          |
| Privilege escalation within containers                   | `no-new-privileges:true`; non-root user (`appuser:1001`); read-only filesystem |
| Network exposure of internal services                    | `internal: true` Docker network; only MCP port 3000 published to host          |
| Brute-force of Valkey credentials                        | Random 32-byte base64 password generated at setup; non-standard TLS port       |

## Network Isolation

The Docker Compose stack uses two networks:

```yaml
networks:
  frontend:
    driver: bridge # app publishes port here
  backend:
    driver: bridge
    internal: true # Valkey-only, no internet
```

This means:

- **Valkey sits on `backend` only** — an `internal: true` network with no external gateway or internet access
- **Valkey has no `ports:` mapping** — it is completely unreachable from the host or any external network
- **The app bridges both networks** — `frontend` for host port publishing, `backend` for Valkey access
- The app container _can_ reach the internet (needed for esm.sh CDN font loading); Valkey _cannot_

Traffic flow:

```
Internet → Host:3000 → excalidraw-mcp (frontend) → (backend, internal) → valkey
                                                         (no internet access)
```

## Encryption in Transit

All communication between the app container and Valkey uses TLS, even though they share an internal Docker network. This is defense in depth — if the Docker network is ever mis-scoped or a container is compromised, data in transit remains encrypted.

### Certificate Setup

Certificates are generated by `docker/valkey/generate-certs.sh` and stored in `docker/valkey/tls/` (gitignored).

| File                                      | Purpose                                  |
| ----------------------------------------- | ---------------------------------------- |
| `ca.key` / `ca.crt`                       | Self-signed CA (4096-bit RSA)            |
| `valkey-server.key` / `valkey-server.crt` | Valkey server certificate (2048-bit RSA) |
| `valkey-client.key` / `valkey-client.crt` | Client certificate for the app           |

The server certificate includes SANs: `DNS:valkey`, `DNS:localhost`, `IP:127.0.0.1`. The client certificate includes SANs: `DNS:excalidraw-mcp`, `DNS:localhost`.

Valkey is configured with `tls-auth-clients optional` — client certificates are validated when presented but not strictly required. The mandatory authentication layer is the ACL password.

### Configuration

Valkey (`docker/valkey/valkey.conf`):

```
port 0          # non-TLS port disabled entirely
tls-port 6379
tls-cert-file /tls/valkey-server.crt
tls-key-file  /tls/valkey-server.key
tls-ca-cert-file /tls/ca.crt
```

App container passes:

```
VALKEY_TLS_ENABLED=true
VALKEY_TLS_CA_CERT=/tls/ca.crt
VALKEY_TLS_CLIENT_CERT=/tls/valkey-client.crt
VALKEY_TLS_CLIENT_KEY=/tls/valkey-client.key
```

### Certificate Rotation

Certificates have a 365-day validity. To rotate:

```bash
# 1. Delete existing certs
rm -rf docker/valkey/tls/*

# 2. Re-generate
bash docker/valkey/generate-certs.sh

# 3. Restart containers (picks up new certs)
docker compose restart
```

## Authentication and Authorization

### Valkey ACL

The default Valkey user is disabled. A dedicated `excalidraw` user enforces least-privilege access:

```
user default off
user excalidraw on >${VALKEY_PASSWORD} ~checkpoint:* ~cp:* +get +set +del +expire +ping +client|setname +client|setinfo +auth resetchannels
```

<!-- prettier-ignore -->
| Restriction | Value |
| --- | --- |
| Allowed key patterns | `checkpoint:*` and `cp:*` only |
| Allowed commands | `get`, `set`, `del`, `expire`, `ping`, `client\|setname`, `client\|setinfo`, `auth` |
| Pub/sub channels | None (`resetchannels`) |
| Password | Random 32-byte base64, set via `VALKEY_PASSWORD` environment variable |

The app cannot read, write, or delete any key outside `checkpoint:*` / `cp:*` — even if the app container is compromised.

### MCP Endpoint

The MCP HTTP endpoint (port 3000) has **no authentication**. Anyone who can reach the host port can call MCP tools. See [What is Not Covered](#what-is-not-covered) for recommendations.

## Encryption at Rest

Valkey checkpoint data is stored in the `valkey-data` Docker volume on the host filesystem. **Valkey does not encrypt data at rest.**

Recommendations by OS:

| Platform | Tool                           | Notes                                    |
| -------- | ------------------------------ | ---------------------------------------- |
| Linux    | LUKS / dm-crypt                | Full disk or volume-level encryption     |
| macOS    | FileVault                      | Encrypts the entire system disk          |
| Any      | Encrypted Docker volume plugin | Third-party; adds operational complexity |

This is the responsibility of the host operator. For most home-lab deployments, OS-level disk encryption is sufficient.

## Secrets Management

### Development / Self-Hosted

Secrets are managed via a `.env` file:

```
PORT=3000
VALKEY_PASSWORD=<random 32-byte base64>
```

The `.env` file is:

- Listed in `.gitignore` — never committed
- Generated by `docker/setup.sh` with a random password on first run:

```bash
RANDOM_PW=$(openssl rand -base64 32)
```

The `.env.example` file (committed) contains only a placeholder:

```
VALKEY_PASSWORD=CHANGE_ME_TO_A_STRONG_RANDOM_PASSWORD
```

### Production

Do not use a `.env` file in production. Use one of:

- **Docker secrets** — `docker secret create valkey_password -` + `secrets:` in compose
- **HashiCorp Vault** — dynamic secrets with lease rotation
- **AWS Secrets Manager / GCP Secret Manager** — managed cloud secrets
- **Environment injection at deploy time** — CI/CD pipeline sets env vars, never stored on disk

Never hardcode secrets in `docker-compose.yml`, `Dockerfile`, or source code.

## Container Hardening

Both containers share the same hardening baseline:

| Control              | App container            | Valkey container         |
| -------------------- | ------------------------ | ------------------------ |
| Read-only filesystem | `read_only: true`        | `read_only: true`        |
| Writable `/tmp`      | `tmpfs: /tmp:size=50M`   | `tmpfs: /tmp:size=10M`   |
| Privilege escalation | `no-new-privileges:true` | `no-new-privileges:true` |
| User                 | `appuser` (UID 1001)     | Valkey default non-root  |
| Memory limit         | 256 MB                   | 192 MB                   |
| CPU limit            | 0.5 cores                | 0.25 cores               |
| Restart policy       | `unless-stopped`         | `unless-stopped`         |

The app image uses a multi-stage build:

1. **Builder stage** (`node:20-alpine`) — installs dependencies, compiles TypeScript, builds the bundle
2. **Runtime stage** (`node:20-alpine`) — copies only `dist/`, runs as `appuser:1001`

No build tools, source files, or dev dependencies are present in the runtime image.

## Certificate Management Reference

```
docker/valkey/tls/        ← gitignored, not committed
├── ca.key                ← CA private key (644)
├── ca.crt                ← CA certificate (644)
├── valkey-server.key     ← Server private key (644)
├── valkey-server.crt     ← Server certificate (644)
├── valkey-client.key     ← Client private key (644)
├── valkey-client.crt     ← Client certificate (644)
└── .gitkeep              ← Keeps directory in git
```

File permissions set by `generate-certs.sh`:

- `*.key` — `644` (world-readable). This is intentional: Docker bind-mount volumes
  preserve host file permissions, and the Valkey container (UID 999) and app container
  (UID 1001) run as different non-root users. Mode `644` allows both to read the keys.
  These are self-signed certificates used only within the Docker internal network.
  For production deployments requiring stricter key isolation, use Docker secrets or
  align container UIDs to a shared GID.
- `*.crt` — `644` (world-readable)

Certificate details:

- CA key: 4096-bit RSA
- Server/client keys: 2048-bit RSA
- Validity: 365 days
- CA subject: `CN=Excalidraw MCP CA/O=Excalidraw MCP`
- Server SAN: `DNS:valkey`, `DNS:localhost`, `IP:127.0.0.1`
- Client SAN: `DNS:excalidraw-mcp`, `DNS:localhost`

## What is Not Covered

These gaps are known and accepted for the self-hosted use case. Address them if deploying in a shared or production environment.

### MCP HTTP endpoint has no authentication or TLS

Anyone who can reach port 3000 on the host can call all MCP tools with no credentials. The endpoint also uses plain HTTP.

**Recommendation:** Place behind a reverse proxy with TLS termination and access control:

```
# Caddy example
excalidraw.yourdomain.com {
    basicauth {
        user <hashed-password>
    }
    reverse_proxy localhost:3000
}
```

Options: Caddy, nginx, Traefik, Tailscale Funnel (zero-config mTLS).

### No rate limiting

The MCP endpoint does not limit request rate. A misconfigured or malicious client can flood the server.

**Recommendation:** Add rate limiting at the reverse proxy layer, or use a middleware like `express-rate-limit`.

### No audit logging

MCP tool calls (which tools were called, with what arguments) are not logged. There is no record of who created or read which checkpoints.

**Recommendation:** Add request logging middleware to `src/server.ts` if auditability is required.

### No intrusion detection

No alerting on unusual access patterns (e.g., scanning unknown key prefixes, repeated auth failures).

**Recommendation:** Enable Valkey's `notify-keyspace-events` and consume events for anomaly detection if needed.

### Widget loads code from esm.sh CDN

The Excalidraw widget loads the Virgil hand-drawn font from `https://esm.sh` at runtime. This means the app container requires internet access and is **not fully air-gapped**. If you need a fully isolated deployment, the font must be vendored into the build.

### `export_to_excalidraw` sends data externally

The `export_to_excalidraw` MCP tool posts diagram JSON to `https://json.excalidraw.com/api/v2/post/`. This sends user diagram data to an external service. For no-data-leak deployments, this tool should be disabled or gated behind an environment variable.

### Valkey data at rest is unencrypted

As noted in [Encryption at Rest](#encryption-at-rest), diagram data in the Docker volume is stored in plaintext on the host filesystem.
